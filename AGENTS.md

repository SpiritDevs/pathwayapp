# CLAUDE.md

## Repo Context

pathwayOS is a minimal web GUI for using coding agents like Codex and Claude. It is an early WIP, so maintainability improvements are welcome when they make the system more reliable, simpler, or more predictable.

Read `AGENTS.md` first and treat it as the source of truth for repo structure, validation requirements, package roles, vendored reference repos, and completion criteria.

## Personal Preferences

### TypeScript

- Never use `any` unless it is genuinely unavoidable or explicitly requested.
- Prefer precise types, schema-derived types, and narrow interfaces over broad escape hatches.
- Keep `packages/contracts` schema-only; do not add runtime logic there.

### Commands

- Do not run dev server commands such as `bun run dev`, `bun run dev:web`, `bun run dev:server`, or `bun run dev:desktop`. Assume the dev server is already running unless explicitly told otherwise.
- Do not run build commands unless specifically asked.
- Prefer checking commands:
  - `vp check`
  - `vp run typecheck`
  - `vp run lint:mobile` when changing native mobile code
  - `vp test` for the built-in Vite+ test command
  - `vp run test` only when the package script named `test` is specifically needed
- If a command is expensive, broad, or likely to take a long time, choose the narrowest useful validation first and explain the tradeoff.

### Package Managers

- This repo uses Bun. Use `bun` for package and script work unless an existing repo command requires `vp`.
- Never use `npm` or `yarn`.
- Do not introduce `pnpm` here unless the repository is intentionally migrated to it.

### Preferred Stack

When uncertain, prefer the stack already present in this repo:

- TypeScript
- Bun
- React
- Vite+
- Effect where the existing package or module already uses it
- Clerk where authentication integrations are needed

Tailwind, Convex, and Vercel are generally preferred for new projects, but do not add them to pathwayOS unless they fit the existing architecture and the change is requested or clearly justified.

### Code Style

- Always strive for concise, simple solutions.
- If a simpler approach solves the problem cleanly, propose or choose it.
- Prefer shared, well-named modules over duplicated local logic.
- Optimize for performance, reliability, and predictable behavior under load, reconnects, restarts, and partial streams.
- Do not edit `.repos/` unless explicitly asked.

### Scope Control

- If asked to do too much at once, stop and say so clearly. Suggest a smaller first slice.
- Preserve unrelated worktree changes. Do not revert or rewrite changes you did not make.
- Read the actual code path before making changes or writing a commit message.

### Computer Use

- If computer use is helpful for completing or verifying work, shell out to gpt-5.5 with Codex for that portion.

## Commit Messages

- Read the relevant code and diff before writing the commit message.
- Use a descriptive title and body.
- Prefix the title with a commit type in brackets, for example:
  - `[fix] Correct provider session reconnect handling`
  - `[style] Tighten sidebar spacing`
  - `[new feature] Add Claude session model picker`
  - `[refactor] Extract shared provider event parsing`

## Picking Models For Workflows And Subagents

Rankings are defaults, not limits. Higher is better. Cost reflects the user's effective cost, not public list price. Intelligence means how hard a problem the model can handle unsupervised. Taste covers UI/UX, code quality, API design, and copy.

| model | cost | intelligence | taste |
| --- | ---: | ---: | ---: |
| gpt-5.5 | 9 | 8 | 5 |
| sonnet-5 | 5 | 5 | 7 |
| opus-4.8 | 4 | 7 | 8 |
| fable-5 | 2 | 9 | 9 |

How to apply:

- These are defaults. You have standing permission to override them.
- Judge the output, not the price tag. If a cheaper model's output does not meet the bar, rerun or redo the work with a smarter model without asking.
- Cost is a tie-breaker only. For anything that ships, prioritize intelligence, then taste, then cost.
- Do not let cost prevent use of the right model. Use cheaper models to gather information and try things before moving important work to a stronger model.
- Bulk or mechanical work, including clear-spec implementation, data analysis, and migrations: use gpt-5.5.
- Anything user-facing, including UI, copy, and API design, needs taste of at least 7.
- Reviews of plans or implementations: use fable-5 or opus-4.8, optionally with gpt-5.5 as an extra independent perspective.
- Never use Haiku.

Mechanics:

- gpt-5.5 is only reachable through the Codex CLI: `codex exec` or `codex review`. The user's `~/.codex/config.toml` defaults to gpt-5.5.
- Use the codex-implementation, codex-review, and codex-computer-use skills where they apply.
- For work those skills do not cover, such as investigation or data analysis, run `codex exec -s read-only` directly with a self-contained prompt.
- Claude models, including sonnet-5, opus-4.8, and fable-5, run via the Agent/Workflow model parameter.

Using gpt-5.5 inside workflows and subagents:

- The model parameter only takes Claude models, so use a wrapper.
- Spawn a thin Claude wrapper agent with `model: "sonnet"` and `effort: "low"` whose prompt tells it to write a self-contained Codex prompt, run `codex exec` via Bash, and return the report.
- Use `schema` on the wrapper to get structured output back.
- Always label these agents with a `gpt-5.5:` prefix, for example `{ label: "gpt-5.5:review-auth" }`. The workflow UI shows the wrapper's Claude model, so the label is the only indication that the real worker is gpt-5.5.
- Codex runs can exceed Bash's 10-minute timeout. Pass an explicit timeout, or run in the background and poll for the report file.
- Parallel gpt-5.5 implementation agents must use `isolation: "worktree"` so Codex edits do not collide in the shared checkout.
- Workflow token budgets only count Claude tokens. Codex work is free and invisible to `budget.spent()`.
